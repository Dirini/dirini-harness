# Interview Skill

사용자의 모호한 아이디어를 깊이 있는 인터뷰를 통해 구체적인 스펙으로 발전시킨다.

## 인터뷰 원칙

### 1. 역방향 요구사항 추출
- 사용자가 "당연하다고 생각해서 안 적는 것들"을 찾아낸다
- 암묵적 가정을 명시적으로 끄집어낸다

### 2. 깊이 있는 질문
- 뻔한 질문 금지 (예: "이름은 뭐로 할까요?")
- 기술 구현, UI/UX, 트레이드오프, 우려 지점을 파고든다

### 3. 객관식 선택지 제시
- "어떻게 할까요?" (X)
- "A 방식 vs B 방식, 어떤 게 맞나요?" (O)

## 인터뷰 영역

다음 영역을 빠짐없이 탐색한다:

| 영역 | 예시 질문 |
|------|----------|
| **목표** | "이걸 통해 최종적으로 얻고 싶은 게 뭔가요?" |
| **사용자** | "누가 이걸 쓰게 되나요? 그들의 기술 수준은?" |
| **범위** | "MVP에 꼭 들어가야 하는 것 vs 나중에 해도 되는 것?" |
| **제약** | "시간/비용/기술 제약이 있나요?" |
| **기존 시스템** | "현재 쓰고 있는 도구/시스템이 있나요?" |
| **성공 기준** | "어떻게 되면 성공이라고 볼 수 있나요?" |
| **우려 지점** | "가장 걱정되는 부분이 있다면?" |
| **트레이드오프** | "속도 vs 품질, 어디에 더 무게를 두나요?" |

## 진행 방식

1. 사용자가 `/interview [주제]` 호출
2. AskUserQuestion 도구로 질문 시작
3. 답변 기반으로 후속 질문 (최소 5-7라운드)
4. 충분히 구체화되면 스펙 파일로 정리

## 완료 조건

다음 조건이 모두 충족되면 인터뷰 종료:
- 목표, 범위, 제약, 성공 기준이 명확해짐
- 사용자가 "이 정도면 됐어" 또는 비슷한 신호를 보냄
- 더 이상 중요한 불확실성이 없음

## 결과물

인터뷰 완료 후:
1. 스펙 문서를 `specs/[주제]-spec.md`로 저장
2. 핵심 결정사항 요약 제공
3. 다음 단계 제안

## Gotchas (자주 발생하는 실수)

- ❌ **첫 질문이 너무 넓음** — "뭘 만들고 싶으세요?"는 나쁜 시작. 사용자가 이미 말한 것에서 구체적 후속 질문으로 시작할 것
- ❌ **5라운드 미만에서 끝내기** — 표면적 답변에서 멈추면 스펙 품질이 낮음. 최소 5라운드, 이상적으로 7라운드
- ❌ **스펙 파일 미생성** — 인터뷰만 하고 결과물을 안 남기면 의미 없음. 반드시 `specs/[주제]-spec.md` 생성
- ❌ **"네/아니오" 질문 연속** — 객관식 선택지(A vs B)로 물어야 깊이 있는 답이 나옴
- ❌ **기존 시스템/맥락 무시** — 사용자가 이미 갖고 있는 도구, 경험, 제약을 먼저 파악하지 않으면 현실과 동떨어진 스펙이 나옴
